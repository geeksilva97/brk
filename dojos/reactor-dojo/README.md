# reactor-dojo

A Claude Code **plugin** that tutors you into building a single-threaded **reactor** (event-loop)
server in Ruby — from a naive blocking accept loop up to an `IO.select` reactor that holds thousands
of connections on one thread — under a deliberate constraint (no web access; you reason from a
mounted Ruby docs bundle and first principles).

Claude Code acts as a **Socratic tutor**: it frames each problem, quizzes you with
free-text questions (scored 1–5), reasons about whether your explanation covers the key concepts, and makes **you type the load-bearing
code** while it handles boilerplate and review.

> Generated by [`dojo-forge`](../../) — this is an example of what the interview produces.

## Install (local dev)

```bash
# One-shot launcher (disables web + prompt suggestions at the CLI level):
./reactor-dojo.sh ~/my-workshop   # create/enter that dir and run the dojo there
# …or the bare claude path:
claude --plugin-dir ./reactor-dojo
# then, in the project where you'll build:
/reactor-dojo:setup     # build the offline docs bundle (stdlib-only — no gems to install)
/reactor-dojo:start     # begin (or resume) at your current step
```

## Commands
- `/reactor-dojo:start` — begin/resume the current step's tutored loop
- `/reactor-dojo:next` — advance after the success check + explain-it-back gate
- `/reactor-dojo:status` — progress + the step ramp
- `/reactor-dojo:hint` — a scoped nudge that never reveals the full spine
- `/reactor-dojo:reveal` — instructor escape hatch: show the reference impl
- `/reactor-dojo:setup` — one-time environment setup (docs bundle)

## The ramp
1. **Blocking accept loop** — feel the freeze: a second client hangs.
2. **Nonblocking accept with IO.select** — wait on the listener without blocking.
3. **The reactor** — one `IO.select` loop, many clients; the hang is gone.
4. **Per-connection state** — buffer partial reads, frame on `\n`.
5. **Backpressure** — writes that would block, parked in the writers set.
6. **Load it** — hold thousands of idle connections on one thread.

## How it's wired
- **Hooks are the jail.** `PreToolUse` denies `WebFetch`/`WebSearch` and external Bash egress, and
  blocks the agent from writing the current step's *spine* file (you type that). `SessionStart`
  resumes you at your step and injects its curriculum. Only `SessionStart` + `UserPromptSubmit` set
  the title (and only when the step changed).
- **State** is per-project: `<project>/.reactor-dojo/progress.json` (keyed to the folder you're in,
  so a new folder starts fresh at Step 1; survives sessions within that project).
- **Stdlib-only.** No gems — `Socket`, `TCPServer`, and `IO.select` are the whole toolkit.
- **Backend model:** the default is a local model via **Ollama** (or llama.cpp) — a true air-gap.
  Pull one (`ollama pull llama3:8b`) and launch with `--model llama3:8b` (the `reactor-dojo.sh`
  wrapper forwards it; Ollama serves on `:11434`); the Anthropic API is an optional easy-mode. Either
  way Claude Code is the harness and the flow is identical.

## Layout
```
.claude-plugin/plugin.json   manifest (no "hooks" field)
reactor-dojo.sh              launch script (one-shot: disables web + prompt suggestions)
skills/tutor/SKILL.md        the 7-beat Socratic loop
commands/                    start, next, status, hint, reveal, setup
hooks/                       session-start, title, guard (jail)
bin/dojo.sh                  state helper (progress.json + steps.tsv)
curriculum/step-01..06.md    the steps (frame/quizzes/spine/checks)
curriculum/steps.tsv         step → title/spine/kind table
curriculum/reference/        instructor reference impls (blocking_echo, reactor, …)
env/docs/build-bundle.sh     builds the offline docs bundle
env/docs/reactor-loop-cheatsheet.md   the GIVEN reactor-loop shape
```
