# agent-dojo Implementation Plan

> **For Hermes:** Use subagent-driven-development skill to implement this plan task-by-task.

**Goal:** Build a Socratic tutoring dojo that teaches how AI agents work by making the learner build one — from a raw LLM call to a full agent loop with tool calling, in TypeScript.

**Architecture:** Claude Code plugin following the c10k-dojo/demonkey pattern. 7-step linear progression. Each step has a spine file the learner types, quizzes with AskUserQuestion, success checks with curl/node, and a fixed next step. Offline jail blocks WebFetch/WebSearch and ad-hoc installs; a docs bundle provides Ollama API reference, Modelfile format, and DuckDuckGo search notes. Weather endpoint (Open-Meteo, no key) and DuckDuckGo are the two tools — added one at a time.

**Tech Stack:** TypeScript + Node.js, Ollama (local LLM), Open-Meteo (free weather API), DuckDuckGo (search via html scraping or duck-duck-go npm).

## Curriculum Design Details

### Step 1 — Ollama + Modelfile

**Frame:** Every AI agent starts with one thing: sending text to a model and getting text back. Before tools, before loops, before any intelligence — there's a raw HTTP call. You'll set up Ollama, create a Modelfile that bakes in a system prompt, and make your first chat() call from TypeScript. The model answers in plain English. That's the floor — everything else builds on top.

**Spine:** `workspace/agent.ts` (~15 lines)
- Create `workspace/Modelfile` with FROM qwen3:8b and a SYSTEM prompt: "You are a helpful assistant. Respond concisely."
- Run `ollama create agent-dojo -f workspace/Modelfile`
- Write `workspace/agent.ts`: import `ollama` from `ollama-ai-provider`, call `ollama.chat({ model: 'agent-dojo', messages: [{ role: 'user', content: 'Hello' }] })`, print the response text.
- Run with `npx tsx workspace/agent.ts`

**Success check:**
1. `ollama create agent-dojo -f workspace/Modelfile` succeeds
2. `npx tsx workspace/agent.ts` prints a plain text response

**Diagnose-quiz:** Why does the Modelfile bake the system prompt into the model instead of sending it with every request?
- ✅ It's pre-pended to every conversation automatically — the model always has the context, no matter how the client calls it. (This is how real tool-calling models receive their tool definitions too.)
- ❌ "Because the API doesn't support system messages." → Wrong — the API supports system messages. The Modelfile just makes it persistent so you don't repeat it.
- ❌ "For security — it can't be overridden." → Wrong — it can be overridden by a system message in the request. It's a default, not a lock.

**Design-quiz:** Your agent.ts sends one message and gets one response. What happens if you send it again in a new call — does the model remember the previous conversation?
- ✅ No — each call is stateless. The model has no memory between calls unless you send the full conversation history. This is why agents maintain a messages array.
- ❌ "Yes, Ollama caches the conversation." → No — each API call is independent. State must be managed by the caller.
- ❌ "It depends on the model." → No — all models in this setup are stateless per-call. Memory is the caller's job.

**Gotchas:**
- Forgetting to run `ollama create` after editing the Modelfile — the model won't pick up changes.
- Using the base model name (qwen3:8b) instead of the custom model name (agent-dojo) — the system prompt won't be active.
- Not handling the promise from `ollama.chat()` — it's async, needs `await`.

### Step 2 — Tool definitions in the Modelfile

**Frame:** A plain text model is useless for real work — it can't fetch weather or search the web. The first step to making it useful is teaching it *how to ask for tools*. You'll add tool definitions to the Modelfile's SYSTEM prompt, and watch the model emit structured JSON instead of plain text when it decides a tool is needed.

**Spine:** `workspace/agent.ts` (modify)
- Update `workspace/Modelfile` SYSTEM prompt to include `[TOOL: get_weather]` and `[TOOL: calculate]` definitions with parameter schemas, plus the RULES block (if you need a tool, respond with a JSON array; if you can answer directly, answer normally).
- Rebuild: `ollama create agent-dojo -f workspace/Modelfile`
- Modify `workspace/agent.ts` to send "What's the weather in Tokyo?" and print the raw response.
- The model should emit `[{"tool": "get_weather", "args": {"location": "Tokyo"}}]` instead of plain text.

**GIVEN black box:** The Modelfile SYSTEM prompt format is provided as a cheatsheet (`open-meteo-cheatsheet.md` gives the weather tool definition; the prompt structure comes from `prompt-emulation/research`). The learner doesn't derive the prompt format — it's a given.

**Success check:**
1. Rebuild succeeds
2. Sending "What's the weather in Tokyo?" returns JSON with `tool: "get_weather"` and `args.location: "Tokyo"`
3. Sending "What is 2+2?" returns a plain text answer (no tool call)

**Diagnose-quiz:** Why does the model emit JSON when it could just write "Let me look that up for you"?
- ✅ Because the SYSTEM prompt explicitly instructs it to respond with a JSON array of tool calls. The model is following the format it was told to use — it's not "deciding" the format, it's complying with instructions.
- ❌ "Because the model has built-in tool calling." → Not in this setup — we're using prompt emulation. There's no formal tool-calling API here. The model follows text instructions.
- ❌ "Because JSON is more efficient than text." → The model doesn't care about efficiency — it outputs JSON because the prompt told it to.

**Design-quiz:** What should happen when the model responds with plain text instead of JSON?
- ✅ That means it decided it could answer directly (like "2+2=4"). The agent should detect this and return the text to the user — no tool execution needed. This is the "direct answer" path.
- ❌ "Try to parse it as JSON anyway." → That would crash or produce garbage. The agent must handle both cases.
- ❌ "Re-prompt the model to use a tool." → No — the model correctly decided no tool was needed. Forcing it would be wrong.

**Gotchas:**
- The model sometimes wraps JSON in backticks or adds explanatory text — the parser needs to handle this.
- `/no_think` in the Modelfile reduces noise but may reduce reasoning quality for complex tool selection.
- If the SYSTEM prompt is too long, some models truncate it — keep tool definitions concise.

### Step 3 — Parse + execute weather

**Frame:** The model asked for the weather — now you have to actually get it. You'll parse the JSON tool call from the model's response, call the Open-Meteo API (free, no key needed), and return the data. This is where the agent stops being a chatbot and starts being... an agent.

**GIVEN black box:** `open-meteo-cheatsheet.md` — the exact Open-Meteo endpoint, parameters, and response format. The learner doesn't need to research the API — it's given.

**Spine:** `workspace/agent.ts` (modify)
- Add a `parseToolCalls(text: string)` function that extracts JSON arrays/objects from the model response (handle backticks, mixed text+JSON).
- Add a `getWeather(location: string)` function that calls Open-Meteo: `fetch(`https://api.open-meteo.com/v1/forecast?latitude=...&longitude=...&current_weather=true`)`. Use a simple lat/lon lookup table or geocoding.
- Wire them: if `parseToolCalls` finds a tool call, execute it and print the result.

**Success check:**
1. `parseToolCalls('Here is the result: [{"tool": "get_weather", "args": {"location": "Tokyo"}}]')` returns the parsed array
2. `getWeather("Tokyo")` returns a JSON object with temperature and condition
3. Running `npx tsx workspace/agent.ts` with "What's the weather in Tokyo?" prints the Open-Meteo data

**Diagnose-quiz:** Why do we need a parser instead of just `JSON.parse`?
- ✅ The model sometimes wraps JSON in markdown backticks, adds explanatory text before/after, or outputs a single object instead of an array. A robust parser handles all these cases — `JSON.parse` alone would choke on surrounding text.
- ❌ "Because JSON.parse is slow." → Performance isn't the issue — robustness is.
- ❌ "Because the API returns YAML." → No, the API returns JSON. The model's text output is the problem.

**Design-quiz:** Why does `getWeather` need a location-to-coordinates mapping?
- ✅ Open-Meteo takes latitude/longitude, not city names. The agent needs to translate "Tokyo" → (35.6762, 139.6503) before calling the API. This mapping is the agent's responsibility — the model just says "Tokyo."
- ❌ "It doesn't — Open-Meteo accepts city names." → Wrong — Open-Meteo's forecast endpoint requires lat/lon.
- ❌ "Because the model outputs coordinates." → No — the model outputs "Tokyo". The translation is your code.

**Gotchas:**
- Open-Meteo requires latitude and longitude — you need a mapping (start with a small object of well-known cities, expand later).
- `fetch` is available in Node 18+ — no need for `node-fetch`.
- The model might output `{"tool": "get_weather", "args": {"location": "Tokyo"}}` (single object) or the array form — handle both.

### Step 4 — Inject + re-prompt

**Frame:** You have the tool result. Now what? You send it back to the model so it can formulate a natural-language answer. This is the "inject" step — the tool result goes into the conversation as a user message, and the model gets a second chance. The model transforms raw JSON into "It's 22°C and partly cloudy in Tokyo."

**Spine:** `workspace/agent.ts` (modify)
- After executing the tool, format the result as a message: `[Tool result: get_weather]\n{"temperature": 22, "condition": "partly cloudy", "humidity": 65}`
- Append it to the messages array as a user message.
- Call `ollama.chat()` again with the full conversation (system + user + assistant tool-call + tool result).
- Print the model's final answer.

**Success check:**
1. Asking "What's the weather in Tokyo?" produces a natural-language answer like "It's currently 22°C and partly cloudy in Tokyo."
2. The messages array now has 4 entries: user question, assistant tool-call JSON, user tool-result, assistant final answer.

**Diagnose-quiz:** Why do we send the tool result as a user message instead of a special "tool" message type?
- ✅ In prompt emulation (no formal tool-calling API), there's no `tool_result` message type. The conversation is just user/assistant turns. We format the result as a user message so the model reads it naturally. Formal APIs have a dedicated message type with ID matching — we don't.
- ❌ "Because user messages have higher priority." → No — the model doesn't prioritize by role here. It's just context.
- ❌ "Because the model can't read assistant messages." → Wrong — the model reads all messages. The format is about matching the prompt pattern the model was trained on.

**Design-quiz:** What would happen if you sent the tool result but didn't include the assistant's original tool-call message in the history?
- ✅ The model would see a user question, then a random piece of JSON about weather, with no context for why. It would be confused — "Why is the user sending me weather data?" The tool-call message is essential context.
- ❌ "It would work fine — the model figures it out." → No — without the assistant's tool call, the result appears out of nowhere.
- ❌ "It would hallucinate." → Not exactly — it would just be confused by contextless data.

**Gotchas:**
- Don't forget to include the assistant's tool-call response in the messages array — without it, the tool result appears out of context.
- The conversation grows with each turn. For long sessions, you'll need truncation or summarization (out of scope for this dojo, but worth noting).

### Step 5 — The agent loop

**Frame:** Steps 2–4 are manual: you call the model, parse the tool call, execute it, inject the result, call again. A real agent does this automatically. The agent loop is a `while` that repeats until the model responds with plain text (no tool call). This is the birth of the agent — three steps become one loop.

**Spine:** `workspace/agent.ts` (modify)
- Wrap the call → parse → execute → inject cycle in a `while (true)` loop.
- If `parseToolCalls` returns empty (model answered directly), break and print the answer.
- If it returns tool calls, execute each one, inject all results, and loop.
- Add a max iteration guard (e.g., 10 turns) to prevent infinite loops.
- Add the interactive readline: prompt the user for input, run the loop, print the answer, repeat.

**Success check:**
1. "What's the weather in Tokyo?" → model calls get_weather → agent executes → injects → model answers in natural language. All automatic, one user prompt.
2. "What is 2+2?" → model answers directly, no tool call, loop exits immediately.
3. A max-iteration guard prevents infinite loops (test with a deliberately confusing prompt if needed).

**Diagnose-quiz:** Why does the loop need a max iteration guard?
- ✅ The model can get stuck in a cycle: calling a tool, getting a result, deciding it needs another tool call, etc. Without a guard, a confused model will loop forever. The guard is a safety net, not a feature — in production you'd add more sophisticated stopping conditions.
- ❌ "Because the API has rate limits." → That's a separate concern. The guard prevents logical loops, not rate limiting.
- ❌ "Because the conversation window is finite." → True, context windows are finite, but the guard prevents logical infinite loops — a different problem.

**Design-quiz:** What's the key difference between this loop and a simple if/else (one tool call)?
- ✅ A single if/else handles one tool call then stops. The loop handles *any number* of tool calls in sequence — the model can call a tool, get a result, realize it needs more information, call another tool, and so on. The loop is what makes it an agent instead of a script.
- ❌ "There's no difference — they do the same thing." → Only if the model always makes exactly one tool call. The loop generalizes.
- ❌ "The loop is for parallel tool calls." → No — parallel calls are a separate feature (Step 6). The loop is for sequential multi-step reasoning.

**Gotchas:**
- The model can emit multiple tool calls in one response (parallel calls) — handle that with a for-loop over the parsed array, even though we only have one tool so far.
- Don't forget to accumulate all tool results before sending back to the model — concatenate them into one user message.
- The `while(true)` pattern is clear but needs the exit condition (no tool calls = break).

### Step 6 — Web search tool

**Frame:** One tool is a demo. Two tools is a decision. The model now has to choose: weather or search? You'll add a DuckDuckGo search tool to the Modelfile, implement the search function, and watch the model decide which tool to call based on the user's question. This is where the agent starts to feel intelligent.

**Spine:** `workspace/agent.ts` (modify)
- Update `workspace/Modelfile` SYSTEM to add `[TOOL: search_web]` definition (parameters: `{"query": "string"}`).
- Rebuild: `ollama create agent-dojo -f workspace/Modelfile`
- Add `searchWeb(query: string)` function using DuckDuckGo html scraping (fetch `https://html.duckduckgo.com/html/?q=encodeURIComponent(query)`, parse the results).
- Extend the tool dispatcher to handle `search_web` in addition to `get_weather`.
- Test: "What's the weather in Tokyo?" calls get_weather. "What happened in the news today?" calls search_web. "Compare weather in Tokyo and London" calls get_weather twice (parallel).

**GIVEN black box:** `duckduckgo-cheatsheet.md` — the DuckDuckGo html endpoint, query format, and how to extract results from the HTML. The learner doesn't need to figure out the API — it's provided.

**Success check:**
1. Rebuild succeeds
2. "What's the weather in Tokyo?" → calls get_weather (not search_web)
3. "What happened in tech news today?" → calls search_web
4. "Compare weather in Tokyo and London" → calls get_weather twice (parallel)
5. The agent correctly routes each question to the right tool

**Diagnose-quiz:** How does the model decide between get_weather and search_web?
- ✅ It matches the user's intent against the tool descriptions. "Weather" maps to get_weather's description; "news" or "latest" maps to search_web. The model reads the descriptions in the SYSTEM prompt and selects semantically — no hard-coded routing.
- ❌ "It uses a keyword filter." → No — there's no if/else routing code. The model reads the descriptions and decides.
- ❌ "It tries both and picks the better result." → No — it decides before executing any tool, based on the description match.

**Design-quiz:** Why does parallel tool calling (two weather calls for two cities) matter for the agent loop?
- ✅ It reduces latency. Instead of: call weather for Tokyo → wait → get result → call weather for London → wait → get result → answer, the agent calls both in one turn, gets both results, and answers. Two round-trips become one.
- ❌ "It doesn't matter — the result is the same." → The result is the same, but the latency doubles without parallel calls.
- ❌ "It's required by the API." → No — the model *chooses* to batch independent calls. It could call them sequentially too.

**Gotchas:**
- DuckDuckGo html scraping can be flaky — add error handling and a timeout.
- The model might call search_web when it could answer directly (e.g., "What is 2+2?") — this is a model quality issue, not a code bug.
- Some queries trigger both tools (e.g., "What's the weather in Tokyo and what happened there recently?") — the model should emit two tool calls in one response.

### Step 7 — Skills as tools

**Frame:** You've built an agent that can call tools. Now you'll make it *learn new behaviors dynamically*. A skill is just another tool — one that reads a markdown file and injects its instructions into the conversation. The model decides when to load a skill based on its description, just like it decides when to call get_weather. This is the final pattern: skills change the model's *style*, tools change its *data*. Both are context injection.

**Spine:** `workspace/agent.ts` (modify)
- Update `workspace/Modelfile` SYSTEM to add `[TOOL: skill]` definition with a description that lists available skills by scanning `skills/*/SKILL.md` frontmatter.
- Add `loadSkill(name: string)` function that reads `skills/${name}/SKILL.md` and strips the YAML frontmatter.
- Extend the tool dispatcher to handle `skill` calls.
- Create `workspace/skills/explain-like-5/SKILL.md` and `workspace/skills/code-review/SKILL.md` as example skills.
- At startup, the agent scans `skills/` and builds the skill tool description dynamically.

**Success check:**
1. "Explain how a database index works, keep it simple" → model calls skill tool with name "explain-like-5" → loads the SKILL.md → answers in simple language
2. "Review this code: def login(user, pw): return db.query(f'SELECT ...')" → model calls skill tool with name "code-review" → loads the SKILL.md → follows the checklist
3. "What's the weather in Tokyo?" → model calls get_weather (not skill) — it routes correctly based on intent

**Diagnose-quiz:** What's the key difference between a tool call (weather, search) and a skill call?
- ✅ A tool call returns *data* — weather JSON, search results. A skill call returns *instructions* that change how the model behaves for the rest of the conversation. Both are context injection, but tools add facts while skills add behavior.
- ❌ "Tools are external, skills are internal." → Both are external — weather is an API call, a skill reads a file. The difference is what they inject.
- ❌ "Skills are more powerful." → Not necessarily — a weather tool can be more impactful than a skill. The difference is data vs. behavior.

**Design-quiz:** Why scan skills at startup instead of hardcoding the list?
- ✅ So the agent can learn new skills without code changes. Drop a new `skills/whatever/SKILL.md` directory and restart — the agent automatically picks it up. This is the same pattern as Claude Code plugins or shell PATH: dynamic discovery beats static lists.
- ❌ "For performance — scanning is faster than hardcoding." → The opposite — scanning is slower. But it's more flexible.
- ❌ "Because the Modelfile can't list skills." → It can — we just choose not to. Dynamic discovery is a design choice for extensibility.

**Gotchas:**
- The skill description in the Modelfile must be rebuilt when skills change — add a rebuild step or make it dynamic at startup.
- The model might try to call a skill that doesn't exist — handle gracefully with an error message.
- Skills are loaded *into the conversation*, not cached between sessions — each new conversation starts fresh.

## Open Questions

1. **Ollama SDK vs raw fetch**: The spine should use `ollama-ai-provider` npm package for cleaner chat calls, or raw `fetch` to `http://localhost:11434/api/chat`? The SDK is simpler but adds a dep. Raw fetch is more educational (shows the actual HTTP). **Recommendation:** Use the SDK for the chat calls (it's thin), but the cheatsheet should show the raw HTTP underneath so the learner understands what's happening.

2. **TypeScript runner**: `npx tsx` or `ts-node`? **Recommendation:** `npx tsx` — zero-config, no tsconfig needed.

3. **City-to-coordinates mapping**: Start with a hardcoded object of ~10 cities, or use a geocoding API? **Recommendation:** Hardcoded map of ~10 major cities as GIVEN scaffold. Geocoding is a distraction.

4. **DuckDuckGo scraping stability**: DDG HTML scraping can break. Should we provide a fallback or use an npm package? **Recommendation:** Use the `duck-duck-go` npm package or similar. Include it in the cheatsheet as the recommended approach, with raw scraping as an alternative.

5. **Skills directory location**: `workspace/skills/` (alongside agent.ts) or a top-level `skills/`? **Recommendation:** `workspace/skills/` — keeps everything the learner touches in one directory.
